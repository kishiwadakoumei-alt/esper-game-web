# ESPER サービス層

## 目的

Flet の画面構築とゲーム状態の変更を分離し、将来の HTTP API / WebSocket 化で同じ操作を再利用できるようにする。

## 構成

### `GameService`

プレイヤー操作と能力処理を担当する。

- 先攻抽選
- ESPER 宣言
- 捨て札と補充
- 能力選択、通常発動、カモフラージュ発動
- 各能力の対象選択と確定
- チャット送信

プリサイエンスは、選択した3枚を指定順で山札の上へ戻してから、通常の手札補充を行う。

### `CpuService`

CPU の判断と1ステップ分の操作を担当する。

- CPU が操作可能な状態かの判定
- 難易度別の捨て札、能力使用、対象選択
- CPU 操作中フラグの開始と終了

待機時間、バックグラウンドスレッド、Flet PubSub による通知は画面側に残す。サービスは同期的に1ステップだけ処理する。

### `RoomService`

対戦部屋のライフサイクルを担当する。

- 対人ルームへの入室
- CPU ルーム作成
- CPU の再戦自動承認
- 再戦要求
- 退出と部屋削除

ルームの保存方式は現行どおりプロセス内辞書であり、永続化や複数プロセス共有は行わない。

## UI側の責務

`ui_views.py` は次の処理だけを担当する。

- Flet コントロールの生成
- 表示用データの読み取り
- イベントからサービスを呼び出す
- PubSub による画面更新通知
- CPU と先攻抽選の待機スレッド起動

ゲーム状態の直接変更は行わない。

## テスト

`tests/test_services.py` で次を検証する。

- 対人入室、CPU 部屋、再戦
- 通常ターン
- タイムリープ
- テレポート
- サイコキネシス
- ヒーリング
- プリサイエンスの山札順
- カモフラージュ経由のプリサイエンス
- CPU の1ステップ操作

実行コマンド:

```bash
python -B -m unittest discover -s tests -v
```
