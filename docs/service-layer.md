# ESPER サービス層

## 目的

ブラウザの画面構築とゲーム状態の変更を分離し、HTTP API、WebSocket、
CPUから同じゲーム操作を再利用できるようにする。

## 構成

### `GameService`

プレイヤー操作と能力処理を担当する。

- 先攻抽選
- ESPER 宣言
- 捨て札と補充
- 能力選択、通常発動、カモフラージュ発動
- 各能力の対象選択と確定
- チャット送信

プリサイエンスは、選択した3枚を指定順で山札の上へ戻してから、通常の手札補充を行う。

### `CpuService`

CPU の判断と1ステップ分の操作を担当する。

- CPU が操作可能な状態かの判定
- 難易度別の捨て札、能力使用、対象選択
- CPU 操作中フラグの開始と終了

サービスは同期的に1ステップだけ処理する。待機時間とバックグラウンドタスクは
`backend/context.py`、リアルタイム通知は
`backend/connection_manager.py` で管理する。

### `RoomService`

対戦部屋のライフサイクルを担当する。

- 対人ルームへの入室
- CPU ルーム作成
- CPU の再戦自動承認
- 再戦要求
- 退出と部屋削除

ルームの保存方式は現行どおりプロセス内辞書であり、永続化や複数プロセス共有は行わない。

### `StateService`

閲覧者ごとに秘匿情報を除外したJSON互換のゲーム状態を生成する。

- 自分と相手で異なる手札情報を返す
- 山札、除外カード、裏向き捨て札を秘匿する
- 能力処理中の一時情報を発動者だけへ返す
- 現在実行可能な操作と表示用選択肢を返す

詳細は `docs/public-state.md` に記載する。

## ブラウザUI側の責務

`frontend/` は次の処理を担当する。

- HTMLによる画面構造
- CSSによる表示とレスポンシブ対応
- 公開ゲーム状態の描画
- HTTP APIへの操作送信
- WebSocketによる状態更新の受信

ゲーム状態の直接変更や、能力結果の確定は行わない。

## テスト

`tests/test_services.py` で次を検証する。

- 対人入室、CPU 部屋、再戦
- 通常ターン
- タイムリープ
- テレポート
- サイコキネシス
- ヒーリング
- プリサイエンスの山札順
- カモフラージュ経由のプリサイエンス
- CPU の1ステップ操作

実行コマンド:

```bash
python -B -m unittest discover -s tests -v
```
